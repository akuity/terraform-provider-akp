# Terraform AKP Provider Version 0.12 Upgrade Guide

Version 0.12.0 of the AKP provider for Terraform introduces three new resources for managing organization-level identity and scoping primitives — `akp_workspace`, `akp_custom_role`, and `akp_api_key` — that previously had to be configured outside of Terraform. This guide covers the upgrade from version 0.11.x to version 0.12.0.

## Provider Version Configuration

→ Before upgrading to version 0.12.0, upgrade to the most recent 0.11.x version of the provider and ensure that
your environment successfully runs [`terraform plan`](https://www.terraform.io/docs/commands/plan.html). You should not
see changes you don't expect or deprecation notices. Also, back up your `.tf` and `.tfstate` files prior to starting the
upgrade process.

Use [version constraints when configuring Terraform providers](https://www.terraform.io/docs/configuration/providers.html#provider-versions).
If you are following that recommendation, update the version constraints in your Terraform configuration and
run [`terraform init -upgrade`](https://www.terraform.io/docs/commands/init.html) to download the new version.

For example, given this previous configuration:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.11.2"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

Update to the latest 0.12.x version:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.12.0"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

## New Features

### `akp_workspace`

Version 0.12.0 introduces an `akp_workspace` resource for managing the workspaces that scope API keys, custom roles, members, and instances within an organization.

```terraform
resource "akp_workspace" "platform" {
  name        = "platform"
  description = "Workspace for platform engineering"
}
```

The default workspace cannot be created or deleted through Terraform — if you need to manage its description, import it first with `terraform import akp_workspace.default <workspace_id>`.

### `akp_custom_role`

The `akp_custom_role` resource manages Akuity custom roles, which attach a Casbin policy that can be referenced from API keys, teams, or OIDC group mappings. A role is either org-scoped (omit `workspace`) or workspace-scoped (set `workspace`).

Each non-empty, non-comment line of `policy` is either a `p, sub, obj, act, resource` rule or a `g, sub, role` grouping. The server enforces scope on the object set: org policies cannot reference workspace-only objects and vice versa.

```terraform
# Org-scoped role granting full management of org API keys.
resource "akp_custom_role" "api_key_manager" {
  name        = "api-key-manager"
  description = "Manage org-level API keys"
  policy      = "p, role:api-key-manager, organization/apikeys, *, *"
}

# Workspace-scoped role granting read-only access to instances in a
# particular workspace.
resource "akp_custom_role" "workspace_instance_viewer" {
  workspace   = akp_workspace.platform.name
  name        = "instance-viewer"
  description = "Read-only access to instances in this workspace"
  policy      = "p, role:instance-viewer, workspace/instances, get, *"
}
```

`name`, `description`, and `policy` can be updated in place. Changing `workspace` triggers replacement.

### `akp_api_key`

The `akp_api_key` resource manages Akuity API keys, either org-scoped (omit `workspace`) or workspace-scoped (set `workspace`). Keys can grant built-in roles, custom roles, or raw action grants, and may optionally expire after a duration.

```terraform
# Org-scoped key with a one-year expiry.
resource "akp_api_key" "ci" {
  description        = "CI deployments"
  expire_in_duration = "8760h"
  permissions = {
    roles = ["member"]
  }
}

# Workspace-scoped key bound to a custom role created by Terraform.
resource "akp_api_key" "platform_admin" {
  workspace          = akp_workspace.platform.name
  description        = "Platform admin key"
  expire_in_duration = "30d"
  permissions = {
    custom_roles = [akp_custom_role.workspace_instance_viewer.id]
  }
}
```

API keys are immutable on the server. Any change to `description`, `permissions`, `expire_in_duration`, or `workspace` triggers replacement, which mints a fresh `secret`.

The generated `secret` is **only available on create or after replacement** — reads from the API cannot recover it. Capture it via a Terraform output or read it from state immediately after the apply that produces it; otherwise you will need to force-replace the resource to obtain a usable secret again.

`permissions` must include at least one of `roles` or `custom_roles`. Workspace-scoped keys additionally accept only the workspace-scoped role names `admin` and `member`; org-scoped keys accept `owner`, `admin`, and `member`.

## Apply the change

After updating the provider version, run `terraform plan` to verify no unintended changes were introduced. None of the new resources affect existing configurations; adopting them is opt-in. If you notice any unexpected differences, please report them as issues.
