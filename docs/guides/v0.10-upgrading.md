# Terraform AKP Provider Version 0.10 Upgrade Guide

Version 0.10.0 of the AKP provider for Terraform is primarily an internal refactoring release focused on reliability and maintainability improvements. It also removes two previously deprecated fields and requires migrating to dedicated resources. This guide will help with the upgrade process and focuses on changes from version 0.9.x to version 0.10.0.

## Provider Version Configuration

-> Before upgrading to version 0.10.0, upgrade to the most recent 0.9.x version (0.9.21) of the provider and ensure that
your environment successfully runs [`terraform plan`](https://www.terraform.io/docs/commands/plan.html). You should not
see changes you don't expect or deprecation notices. Also, back up your `.tf` and `.tfstate` files prior to starting the
upgrade process.

Use [version constraints when configuring Terraform providers](https://www.terraform.io/docs/configuration/providers.html#provider-versions).
If you are following that recommendation, update the version constraints in your Terraform configuration and
run [`terraform init -upgrade`](https://www.terraform.io/docs/commands/init.html) to download the new version.

For example, given this previous configuration:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.9.21"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

Update to the latest 0.10.0 version:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.10.0"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

## Breaking Changes

Version 0.10.0 removes the following deprecated fields:

| Removed field | Removed from | Replacement |
| --- | --- | --- |
| `argocd.spec.instance_spec.ip_allow_list` | `akp_instance` resource and `data.akp_instance` | `akp_instance_ip_allow_list` resource |
| `kargo.spec.kargo_instance_spec.default_shard_agent` | `akp_kargo_instance` resource and `data.akp_kargo_instance` | `akp_kargo_default_shard_agent` resource |

### Migrate `argocd.spec.instance_spec.ip_allow_list`

The `argocd.spec.instance_spec.ip_allow_list` field has been removed from the `akp_instance` resource and the `akp_instance` data source.

Use the `akp_instance_ip_allow_list` resource to manage Argo CD instance IP allow list entries. The `backend_ip_allow_list_enabled` field remains on `akp_instance` and should stay there if you use the allow list. Version 0.10.0 does not provide a separate replacement data source for the allow list; import the resource if Terraform needs to adopt existing entries.

Previous configuration:

```terraform
resource "akp_instance" "example" {
  name = "example"
  argocd = {
    spec = {
      version = "v3.3.3-ak.77"
      instance_spec = {
        backend_ip_allow_list_enabled = true
        ip_allow_list = [
          {
            ip          = "192.168.1.0/24"
            description = "Office network"
          },
          {
            ip          = "10.0.0.0/8"
            description = "VPN"
          },
        ]
      }
    }
  }
}
```

Updated configuration:

```terraform
resource "akp_instance" "example" {
  name = "example"
  argocd = {
    spec = {
      version = "v3.3.3-ak.77"
      instance_spec = {
        backend_ip_allow_list_enabled = true
      }
    }
  }
}

resource "akp_instance_ip_allow_list" "example" {
  instance_id = akp_instance.example.id
  entries = [
    {
      ip          = "192.168.1.0/24"
      description = "Office network"
    },
    {
      ip          = "10.0.0.0/8"
      description = "VPN"
    },
  ]
}
```

If the instance already has allow list entries configured remotely, import them into the new resource before running `terraform apply`:

```bash
terraform import akp_instance_ip_allow_list.example <instance_id>
```

This import step is important for existing configurations. Creating `akp_instance_ip_allow_list` against an instance that already has those IPs configured will be treated as a duplicate configuration.

### Migrate `kargo.spec.kargo_instance_spec.default_shard_agent`

The `kargo.spec.kargo_instance_spec.default_shard_agent` field has been removed from the `akp_kargo_instance` resource and the `akp_kargo_instance` data source.

Use the `akp_kargo_default_shard_agent` resource to manage the default shard agent for a Kargo instance.

Previous configuration:

```terraform
resource "akp_kargo_instance" "example" {
  name = "example"
  kargo = {
    spec = {
      version = "v1.9.5-ak.0"
      kargo_instance_spec = {
        default_shard_agent = "my-agent"
      }
    }
  }
}
```

Updated configuration:

```terraform
resource "akp_kargo_instance" "example" {
  name = "example"
  kargo = {
    spec = {
      version = "v1.9.5-ak.0"
      kargo_instance_spec = {}
    }
  }
}

resource "akp_kargo_default_shard_agent" "example" {
  kargo_instance_id = akp_kargo_instance.example.id
  agent_id          = akp_kargo_agent.my_agent.id
}
```

If a default shard agent is already configured remotely and you want Terraform to adopt it without changing it, import the new resource before running `terraform apply`:

```bash
terraform import akp_kargo_default_shard_agent.example <kargo_instance_id>/<agent_id>
```

## Internal Improvements

Version 0.10.0 includes a significant internal refactoring of the provider's type conversion system and resource lifecycle management. While these changes do not affect the provider's external API, they improve reliability and consistency across all resources:

- Unified resource lifecycle handling for consistent Create, Read, Update, Delete, and Import operations across all resources.
- Rewritten type conversion system between Terraform state and Akuity Platform API models, reducing inconsistent state issues.
- Improved plan modifier behavior for optional and computed fields.

## akp_kargo_default_shard_agent (Data Source)

Version 0.10.0 also introduces a new data source `akp_kargo_default_shard_agent` that allows you to read the default shard agent configuration for a Kargo instance:

```terraform
data "akp_kargo_default_shard_agent" "example" {
  kargo_instance_id = akp_kargo_instance.example.id
}

output "default_agent_id" {
  value = data.akp_kargo_default_shard_agent.example.agent_id
}
```

### Schema

- `kargo_instance_id` (String, Required) - The ID of the Kargo instance
- `agent_id` (String, Read-Only) - The ID of the Kargo agent set as the default shard agent

## Apply the change

After removing the deprecated fields from your configuration, adding the replacement resources, and importing existing remote configuration where needed, run `terraform plan` to verify no unintended changes were introduced. Aside from the explicit migration to the replacement resources, the internal refactoring should produce identical plan output compared to version 0.9.21. If you notice any unexpected differences, please report them as issues.
