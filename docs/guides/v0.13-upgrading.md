# Terraform AKP Provider Version 0.13 Upgrade Guide

Version 0.13.0 of the AKP provider for Terraform introduces two new resources for managing organization teams and workspace access — `akp_team` and `akp_workspace_member` — that previously had to be configured outside of Terraform. This guide covers the upgrade from version 0.12.x to version 0.13.0.

## Provider Version Configuration

→ Before upgrading to version 0.13.0, upgrade to the most recent 0.12.x version of the provider and ensure that
your environment successfully runs [`terraform plan`](https://www.terraform.io/docs/commands/plan.html). You should not
see changes you don't expect or deprecation notices. Also, back up your `.tf` and `.tfstate` files prior to starting the
upgrade process.

Use [version constraints when configuring Terraform providers](https://www.terraform.io/docs/configuration/providers.html#provider-versions).
If you are following that recommendation, update the version constraints in your Terraform configuration and
run [`terraform init -upgrade`](https://www.terraform.io/docs/commands/init.html) to download the new version.

For example, given this previous configuration:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.12.0"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

Update to the latest 0.13.x version:

```terraform
terraform {
  required_providers {
    akp = {
      source  = "akuity/akp"
      version = "~> 0.13.0"
    }
  }
}

provider "akp" {
  # Configuration options
}
```

## New Features

### `akp_team`

Version 0.13.0 introduces an `akp_team` resource for managing organization teams. A team groups users and can be granted organization-level custom roles; teams can also be added to workspaces via `akp_workspace_member`.

```terraform
resource "akp_team" "platform" {
  name        = "platform"
  description = "Platform engineering team"
}

# Team granted an org-level custom role.
resource "akp_custom_role" "api_key_manager" {
  name        = "api-key-manager"
  description = "Manage org-level API keys"
  policy      = "p, role:api-key-manager, organization/apikeys, *, *"
}

resource "akp_team" "operators" {
  name         = "operators"
  description  = "On-call operators"
  custom_roles = [akp_custom_role.api_key_manager.id]
}
```

A team is identified by `name`, which is its natural key — the API has no separate team ID. Changing `name` therefore triggers replacement. `description` and `custom_roles` can be updated in place.

`custom_roles` takes the **IDs** of org-scoped custom roles (the `id` attribute, e.g. `akp_custom_role.api_key_manager.id`), not their names. Only org-scoped roles are accepted; passing a workspace-scoped role's ID is rejected by the server. Granting custom roles requires the custom roles feature to be enabled for your organization.

`create_time` and `member_count` are read-only, server-assigned values.

### `akp_workspace_member`

The `akp_workspace_member` resource adds a member to a workspace and grants it a `role` on that workspace. A member is either a user (identified by `user_email`) or a team (identified by `team_name`); exactly one of the two must be set.

```terraform
resource "akp_workspace" "platform" {
  name = "platform"
}

# Add an existing user to the workspace as an admin.
resource "akp_workspace_member" "alice" {
  workspace  = akp_workspace.platform.name
  role       = "admin"
  user_email = "alice@example.com"
}

# Add a whole team to the workspace.
resource "akp_team" "platform" {
  name = "platform"
}

resource "akp_workspace_member" "platform_team" {
  workspace = akp_workspace.platform.name
  role      = "member"
  team_name = akp_team.platform.name
}
```

`role` accepts `member` or `admin` and is the only attribute that can be updated in place. `workspace`, `user_email`, and `team_name` all trigger replacement when changed.

When adding a user, `user_email` must reference a **user that already exists** in the organization — this resource grants workspace access, it does not invite or create users. The email is matched case-insensitively and is stored in canonical (lowercase) form on the server, so prefer writing it in lowercase in your configuration.

Workspace members can be imported using `<workspace_name>/<member_id>`:

```shell
terraform import akp_workspace_member.platform_team platform/wm-abc123
```

After import, the resource recovers the role and member identity; write configuration whose member identifier (`user_email` or `team_name`) matches the imported member. The `id` (membership record ID) and `workspace_id` are read-only, server-assigned values.

Managing workspace members requires the workspaces feature to be enabled for your organization.

## Apply the change

After updating the provider version, run `terraform plan` to verify no unintended changes were introduced. None of the new resources affect existing configurations; adopting them is opt-in. If you notice any unexpected differences, please report them as issues.
